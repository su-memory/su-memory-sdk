# su-memory SDK — 本地优先训练路线重评估

> **评估日期**: 2026-04-25  
> **评估环境**: Apple M5 Pro / 18 核 (6 Super + 12 Performance) / 48 GB 统一内存 / macOS 26.5  
> **前置文档**: v3.5.0 实施计划 / v3.7.0 可行性评估

---

## 〇、战略转向：从"云端训练"到"本地实用"

### 原路线 (v3.7.0 可行性评估)

| 维度 | 原计划 | 问题 |
|---|---|---|
| 模型 | 训练 1B 专属记忆模型 (from scratch) | 需要 A100/H100，用户不可及 |
| 框架 | PyTorch + DeepSpeed | 不在 Mac 上优化 |
| 数据 | 100K-500K QA 对 | 云端训练时合理，本地需更精 |
| 部署 | pip install + 下载模型 | 用户感知重 |

### 修正路线：本地优先三原则

1. **移动实用性**: 训练与推理均在用户 Mac 上完成，零外部依赖
2. **增量 LoRA**: 不训新模型，用 QLoRA 在 Qwen2.5-1.5B 上叠加记忆能力
3. **数据精炼**: M5 合成质量 > 数量，目标 10K-50K 高信噪比 QA 对

## 一、可行性基准测试

### 1.1 MLX 矩阵乘法吞吐量 (M5 Pro 实测)

| 模型规模 | hidden_dim | matmul 延迟 | 每层参数 | 评估 |
|---|---|---|---|---|
| 0.5B (LoRA r=32) | 512 | 2.02ms | ~2M | 极快 |
| 0.5B (LoRA r=64) | 512 | 0.16ms | ~2M | 极快 |
| 1.5B (LoRA r=32) | 1536 | 0.32ms | ~16M | **推荐** |
| 1.5B (LoRA r=64) | 1536 | 0.41ms | ~16M | **推荐** |
| 3B (LoRA r=32) | 2048 | 0.46ms | ~29M | 可行，较慢 |

> 注: 第二次运行比第一次快 10x+，因为第一次包含 MLX JIT 编译预热。实际训练时编译仅发生在第一步。

### 1.2 QLoRA 4-bit 内存占用 (48GB 环境)

| 模型 | BF16 | 4-bit 基础 | LoRA 适配器 | 训练峰值 | 可用余量 |
|---|---|---|---|---|---|
| Qwen2.5-0.5B | 1.0 GB | 0.25 GB | ~20 MB | ~3 GB | 45 GB ✅ |
| **Qwen2.5-1.5B** | **3.0 GB** | **0.75 GB** | **~100 MB** | **~6 GB** | **42 GB** ✅ |
| Qwen2.5-3B | 6.0 GB | 1.5 GB | ~200 MB | ~10 GB | 38 GB ✅ |
| Qwen2.5-7B | 14.0 GB | 3.5 GB | ~400 MB | ~20 GB | 28 GB ✅ |

> 结论: 48GB 统一内存在 QLoRA 4-bit 下足以训练到 7B，**1.5B 是最佳甜点**（训练时间/模型能力/内存余量平衡）。

### 1.3 训练时间估算 (1.5B QLoRA)

```
假设:
  - 序列长度: 512 tokens
  - Batch size: 4
  - 每步时间: ~0.3s (正向+反向+优化器, 含编译后稳定态)
  - 每 QA 对: 2 条消息 (cause + effect)
  - Epochs: 3

  10K QA 对 → 20K messages / 4 = 5,000 steps × 3 epochs = 15,000 steps
  15,000 × 0.3s = 4,500s ≈ 1.3 小时

  30K QA 对 → 45,000 steps × 0.3s = 13,500s ≈ 3.8 小时

  50K QA 对 → 75,000 steps × 0.3s = 22,500s ≈ 6.3 小时

  100K QA 对 → 150,000 steps × 0.3s = 45,000s ≈ 12.5 小时
```

| 数据规模 | 训练时间 | 使用场景 |
|---|---|---|
| 10K QA | ~1.3 小时 | 快速迭代/POC |
| 30K QA | ~3.8 小时 | **推荐: 平衡质量与速度** |
| 50K QA | ~6.3 小时 | 隔夜训练 |
| 100K QA | ~12.5 小时 | 周末训练 |

## 二、修正后的 v3.6.0 方案

### 2.1 架构变更

```
原方案 (云端)                       修正方案 (本地 M5 Pro)
─────────────────                   ─────────────────────
训练 1B 模型 from scratch      →   QLoRA 在 Qwen2.5-1.5B-Instruct 上叠加
PyTorch + DeepSpeed + 8×H100  →   MLX + MLX-LM + M5 Pro 统一内存
100K-500K Reflection QA 对     →   10K-50K 精炼 QA 对 (质量优先)
多 GPU 数据并行                →   单设备统一内存 (零拷贝)
TIES 模型合并 (多租户)         →   LoRA 适配器合并 (轻量, ~100MB/租户)
模型分发: 下载 2GB 权重        →   用户下载 ~100MB LoRA + 已有 Qwen
```

### 2.2 MLX-LM LoRA 训练流程


### 2.3 关键设计决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 基础模型 | Qwen2.5-1.5B-Instruct (MLX 4-bit) | MEMO 已验证 Qwen 家族在记忆任务上有效 (Table 4) |
| 微调方法 | QLoRA (rank=64, alpha=128) | 仅训练 ~100M 参数 (1.5B 的 6.7%) |
| 训练框架 | MLX + MLX-LM | Apple Silicon 原生，统一内存零拷贝 |
| 数据格式 | su-memory 定制 JSONL | `{"cause": "...", "effect": "...", "energy": "enhance", "bf": 3.5}` |
| 损失函数 | SFT + 能量一致性正则 | 标准交叉熵 + 生克关系约束 |
| 推理集成 | `SuMemoryLitePro.predict_causal()` | 零侵入现有 API，新增 `use_model=True` 参数 |
| 模型存储 | `~/.su-memory/models/lora_adapters.safetensors` | ~100MB, 可 git-lfs 或 pip package_data |

## 三、对 v3.5.0 M5 的影响

M5 (Reflection QA 合成) 的角色从"为云端训练准备海量数据"转变为"为本地训练准备精炼数据"。

### 3.1 M5 数据质量控制升级

| 原 M5 设计 | 修正后 |
|---|---|
| max_pairs=500 | max_pairs=200 (更精) |
| min_confidence=0.3 | min_confidence=0.4 (更高门槛) |
| BayesianCausal 筛选 | **新增**: 能量一致性筛选 |
| 无数据多样性检查 | **新增**: 五行覆盖均衡检查 |

### 3.2 新增: 训练数据质量报告

```python
# _reflection_synthesizer.py 新增方法

def training_data_report(self, pairs: List[SynthesizedQAPair]) -> Dict:
    """
    生成训练数据质量报告。
    
    Returns:
        {
            "total_pairs": int,
            "avg_confidence": float,
            "energy_distribution": {"enhance": N, "suppress": M, ...},
            "diversity_score": float,  # 五行覆盖均衡度
            "ready_for_training": bool,  # 是否达到训练门槛
        }
    """
```

### 3.3 训练就绪门槛

| 指标 | 门槛 | 说明 |
|---|---|---|
| 总 QA 对数 | ≥ 5,000 | 太少则过拟合风险高 |
| 平均置信度 | ≥ 0.40 | BayesianCausal 后验筛选 |
| 五行覆盖 | 每组 ≥ 500 | 避免单一能量类型过拟合 |
| 多样性得分 | ≥ 0.60 | 词汇/主题多样性 |

## 四、对 v3.5.0 M4 的影响

M4 (噪声梯度验证) 的退出决策矩阵需要增加一个维度：如果 noise_robustness ≥ 0.60（检索范式可接受），本地训练作为**增值能力叠加**而非替代；如果 < 0.40，本地训练作为**必需补丁**。

| noise_robustness | 原决策 | 修正决策 (本地训练) |
|---|---|---|
| ≥ 0.80 🟢 | v3.6.0 优先级降 | 本地训练作为可选增强 (LoRA 适配器) |
| 0.60–0.80 🟡 | v3.6.0 按计划 | 本地训练作为推荐升级路径 |
| 0.40–0.60 🟠 | v3.6.0 升 P0 | 本地训练作为紧急修复 |
| < 0.40 🔴 | 跳过 M5/M6 | 本地训练作为生存必需 |
