| 维度 | su-memory SDK（本计划） | MCI World Model | MCI-SDK |

| **合规层** | ✅ 本计划新增（PHI/审计/删除权） | ❌ 不涉及 | ❌ 不涉及 |

### 关键区隔约束（本计划必须遵守）

**约束 1：因果推理的边界——"检索因果" vs "世界因果"**

su-memory 内部有一套轻量因果图（`_sys/_causal_engine.py`，五行能量关系驱动），
MCI World Model 有完整的 Pearl 三层因果引擎（do-calculus/反事实/PC/GES/LiNGAM）。

**边界划分**：
- su-memory 的因果图 **仅用于记忆检索排序**（"哪条记忆和哪条关联更强"）
- su-memory **不做** do-calculus、反事实推理、因果发现——这是 World Model 的职责
- 本计划 P1-S1 "CausalDAG 医疗因果注入"指的是给**检索因果图**注入医疗边（"华法林记忆"和"维K记忆"关联更强），**不是**做"如果停药会怎样"的反事实推理

**修订**：P1-S1 重命名为 **"医疗关联图注入"**，明确是检索关联增强而非因果推断。

**约束 2：贝叶斯推理的边界——"检索置信度" vs "世界概率模型"**

su-memory 和 World Model 都有 `bayesian_network.py`（代码几乎相同，从同一源码分叉）。

**边界划分**：
- su-memory 的贝叶斯 **仅用于记忆检索置信度评估**（"这条记忆有多可靠"）
- World Model 的贝叶斯 **用于世界状态概率预测**（"患者状况恶化的概率"）
- 本计划 P1-S2 "贝叶斯临床概率模型"**降级为可选**：如果需要真正的临床风险概率预测（如"再喂养综合征 P=0.85"），应通过 World Model 做，su-memory 仅提供记忆证据

**修订**：P1-S2 改为 **"记忆置信度增强"**——用贝叶斯评估记忆可靠性，不做临床风险预测。临床风险预测委托 World Model。

**约束 3：增量学习的边界——"检索优化" vs "世界模型训练"**

**边界划分**：
- su-memory 的增量学习 **仅优化检索排序权重**（好评方案排名↑）
- World Model 的学习 **更新世界预测模型参数**
- 本计划 P1-S3 不变，但明确：训练目标是检索排序，不是世界状态预测

**约束 4：代码物理隔离——禁止共享源码文件**

当前 su-memory 和 World Model 的 `_sys/` 目录有 25 个同名文件（从同一源码分叉）。
两者已声明独立仓库，**不得在 su-memory 中 import mci_world_model**，反之亦然。

**约束 5：MCI-SDK 的记忆委托关系**

MCI-SDK（Agent 编排引擎）依赖 su-memory（`su-memory>=3.5.0`），其 MemoryLayer 模块
将记忆操作委托给 su-memory。本计划的 `su_memory.clinical` 适配层不得破坏 MCI-SDK
的 `MemoryLayer` 接口契约。

### 修订后的本计划范围（精简版）

基于区隔约束，本计划做了以下调整：

| 原 P1-Sp | 原名称 | 修订后 | 理由 |
|----------|--------|--------|------|
| P1-S1 | CausalDAG 医疗因果注入 | → **医疗关联图注入**（检索关联增强，非因果推断） | 因果推断归 World Model |
| P1-S2 | 贝叶斯临床概率模型 | → **记忆置信度增强**（检索可靠性评估，非临床预测） | 临床预测归 World Model |
| P1-S3 | 增量学习飞轮 | ✅ 不变（检索排序优化） | 明确是检索目标 |
| P1-S4 | 模式提炼 | ✅ 不变（群体记忆模式发现） | 不涉及世界模型 |

**净效果**：su-memory 聚焦"医疗记忆的存储/检索/合规"，不越界做因果推断和世界预测。
如果临床场景需要"如果给这个患者用华法林会怎样"的反事实推理，由 World Model 负责，
su-memory 只负责"提供该患者的全部相关记忆作为 World Model 的输入"。

## 第零层：第一性原理推导

### 原点问题：「医疗智能体为什么需要记忆？」

剥离所有技术术语，回到最根本的：

> **一个临床医生在决策时，依赖什么信息？**

```
                 ┌──────────────────────────────────────┐
                 │          临床决策的信息依赖            │
                 └──────────┬───────────────────────────┘
                            │
           ┌────────────────┼────────────────┐
           │                │                │
     ① 患者是谁        ② 发生了什么     ③ 规则说什么
     （纵向身份）      （纵向事件流）    （横向知识）
           │                │                │
    基础信息/过敏/禁忌   诊疗时间线/检验趋势   指南/药物交互/循证
```

**推导结论**：医疗记忆引擎必须同时承载三种记忆——

| 记忆类型 | 本质 | 当前 SDK 支持 | 差距 |
|----------|------|---------------|------|
| **患者纵向记忆** | "这个人的完整诊疗历史" | ✅ recall + patient_id 隔离 | 缺诊疗周期/趋势 |
| **事件因果记忆** | "方案A→腹胀→方案B"的因果链 | ⚠️ 有 CausalDAG 但仅用于检索排序 | 缺事件间因果关系建模 |
| **领域知识记忆** | "华法林+维K=拮抗" | ❌ 完全缺失 | 零医疗知识 |

### 第二层推导：「医疗场景对记忆引擎的约束是什么？」

不是所有约束都同等重要。用第一性原理排序：

| 约束 | 为什么是刚需 | 不可妥协程度 |
|------|-------------|-------------|
| **零延迟响应** | 医生在门诊等 AI 回复，>3秒不可接受 | 🔴 绝对 |
| **断网可用** | 医院内网常态隔离，不能依赖外部 API | 🔴 绝对 |
| **数据隔离** | 患者A 绝对不能看到患者B 的数据 | 🔴 绝对（法律） |
| **可审计** | 每条建议必须能追溯到依据 | 🔴 绝对（法规） |
| **可删除** | 患者有权要求删除所有数据 | 🔴 绝对（个保法） |
| **可解释** | "为什么推荐这个？"必须有推理链 | 🟡 强需求 |
| **因果推理** | 过敏→禁忌的自动推导 | 🟡 强需求 |
| **多模态** | 检验值/影像/文本混合 | 🟢 可演进 |

### 第三层推导：「SDK 已有的能力中，哪些被医疗场景浪费了？」

SDK 有大量高级能力，但当前医疗场景完全没有利用：

| SDK 能力 | 代码行数 | 医疗场景利用 | 浪费原因 |
|----------|----------|-------------|----------|
| CausalDAG 因果图 | ~2000行 | ❌ 仅检索排序 | 没有医疗因果知识注入 |
| 贝叶斯推理网络 | ~2400行 | ❌ 完全未用 | 没有临床概率模型 |
| 代数层 GF(2)³/群论 | ~3000行 | ❌ 完全未用 | 纯数学，无医疗映射 |
| 五行能量系统 | ~4000行 | ❌ 仅分类标签 | 中医映射未接通 |
| 增量学习 | 937行 | ❌ 完全未用 | 没有反馈训练循环 |
| 分层存储 hot/warm/cold | 600行 | ❌ 未激活 | 患者冷热数据策略未配 |
| 模式提炼 distill_patterns | ✅ | ❌ 完全未用 | 没有临床模式数据 |
| LangChain 适配器 | ✅ | ❌ 完全未用 | Agent 框架未接通 |

**核心洞察**：SDK 的 60K+ 行代码中，超过 **70% 的能力在医疗场景下沉睡**。这不是"要加什么"的问题，而是"如何激活已有能力"的问题。

### Phase 1 · 引擎激活（2 周 · P0）

**第一性原理**：不写新引擎，激活 SDK 已有但沉睡的能力。

#### P1-S1 · CausalDAG 医疗因果知识注入（3 天）

**现状**：`_sys/_causal_engine.py`（CategoryCausalEngine）有完整的因果图推理，但图的边权重基于五行能量关系（wood/fire/earth/metal/water），与医疗无关。

**方案**：注入医疗因果边，不改引擎代码——

```
当前因果图边： wood enhances fire, fire suppresses metal, ...
新增因果边：   warfarin → suppresses → vitamin_k_absorption
              protein_deficiency → causes → hypoalbuminemia
              peanut_allergy → contraindicates → peanut_protein_formula
```

**交付**：
- `src/su_memory/clinical/causal_kb.py`
  - `MedicalCausalGraph` 类：加载医疗因果三元组
  - `inject_into_engine(memory_client)` 方法：把医疗边注入 CausalDAG
  - 种子数据（20+ 条验证过的临床因果链）：
    - 药物-营养交互（华法林/维K、甲氨蝶呤/叶酸…）
    - 营养缺乏-临床表现（低白蛋白→水肿、缺铁→贫血…）
    - 过敏-禁忌（花生过敏→禁用相关制剂…）
- `query_multihop("服用华法林要注意什么营养问题")` 自动走因果图推导

**验收**：
- `inject_into_engine()` 后，多跳查询能推导出"华法林→维K拮抗→限制深色蔬菜"
- 因果推理链可解释（`explain_multihop` 返回完整推导路径）
- 断网环境下不阻塞（种子数据内置，不依赖外部 API）

**工作量**: 3 天

---

#### P1-S2 · 贝叶斯推理临床概率模型（3 天）

**现状**：`_sys/bayesian_reasoning.py`（BayesianReasoningSystem，2400+行）有完整的贝叶斯网络推理能力，但没有任何临床概率模型。

**方案**：构建临床概率贝叶斯网络——

```
               营养不良风险
              /     |      \
         白蛋白低  BMI低  进食量少
           |        |       |
         感染风险  跌倒风险  愈合延迟
```

**交付**：
- `src/su_memory/clinical/bayesian_models.py`
  - `NutritionRiskNetwork`：营养不良→并发症的概率网络
  - `RefeedingRiskNetwork`：再喂养综合征风险预测
  - `PressureInjuryRiskNetwork`：压疮营养风险
  - 每个网络含：节点定义 + 条件概率表(CPT) + 证据推理接口
  - `predict_risk(patient_id) → RiskAssessment`：从患者记忆提取证据→推理风险

**验收**：
- 写入患者记忆（白蛋白=28, BMI=18.5, 进食量=50%），`predict_risk` 返回"营养不良高风险(P=0.85)"
- 贝叶斯推理可解释（返回各节点的后验概率）
- 与 `recall()` 协同：记忆中有的证据自动输入网络

**工作量**: 3 天

---

#### P1-S3 · 增量学习飞轮激活（2 天）

**现状**：`_sys/_incremental_learning.py`（937行）有完整的增量学习框架，但 feedback_loop 只写入了记忆，没有训练模型。

**方案**：把 feedback_loop 的"评分≥4的反馈"作为增量学习的训练信号——

```
高质量反馈（rating≥4）→ 增量学习更新权重 → 下次检索排序优化
低质量反馈（reject）  → 负样本降权       → 下次避免类似推荐
```

**交付**：
- `src/su_memory/clinical/feedback_trainer.py`
  - `ClinicalFeedbackTrainer`：桥接 feedback_loop → 增量学习
  - `train_from_feedback(memory_id, rating, action)` 方法
  - 自动调整检索权重（高评分方案的关联记忆权重↑）
- 营养系统 `feedback_loop.py` 的 `_write_feedback_to_memory` 中追加训练调用

**验收**：
- 模拟 10 条"高蛋白方案+好评"反馈后，查询"营养不良方案"时高蛋白方案排名↑
- 模拟 3 条"低蛋白方案+差评"后，低蛋白方案排名↓
- 训练过程 < 100ms（不阻塞 feedback_loop）

**工作量**: 2 天

---

#### P1-S4 · 模式提炼：临床决策模式发现（2 天）

**现状**：SDK 有 `distill_patterns()` 和 `extract_rules()` 方法，但从未在临床数据上运行过。

**方案**：从大量患者记忆中自动提炼临床决策模式——

```
1000个患者的记忆 → distill_patterns() → "糖尿病肾病患者普遍适合低蛋白高热量方案"
                                     → "老年患者对高蛋白方案依从性差"
```

**交付**：
- `src/su_memory/clinical/pattern_miner.py`
  - `mine_clinical_patterns(diagnosis=None, min_support=10)` 方法
  - 输出：`ClinicalPattern` 列表（条件→结论+支持度+置信度）
  - `pattern_to_rule(pattern) → ClinicalRule`：模式转化为可执行规则
- 定期任务接口（供 CI/定时任务调用）

**验收**：
- 在 50+ 条模拟患者记忆上运行，输出至少 2 个有意义的临床模式
- 模式可注入因果图（强化 P1-S1 的因果边权重）

**工作量**: 2 天

## 第七层：与 v1.0 计划的差异（为什么 v2.0 更好）

| 维度 | v1.0 计划 | v2.0（第一性原理） |
|------|-----------|-------------------|
| 思路 | "要加什么新功能" | "要激活什么已有能力" |
| 新代码量 | ~3000 行新写 | ~1500 行新写 + 激活 12000 行已有 |
| 因果推理 | 新写药物交互查询 | 注入已有 CausalDAG（2000 行复用） |
| 风险预测 | 不涉及 | 激活已有贝叶斯网络（2400 行复用） |
| 自学习 | 不涉及 | 激活已有增量学习（937 行复用） |
| 模式发现 | 不涉及 | 激活已有 distill_patterns |
| 医疗价值深度 | 知识查询 + 合规 | 因果推理 + 概率预测 + 自学习 + 模式发现 |
| 核心原则 | 功能堆叠 | 第一性原理：激活 > 新建 |

**核心差异**：v2.0 不是"往 SDK 上贴医疗功能"，而是"把 SDK 已有的 70% 闲置能力通过医疗知识激活"。
