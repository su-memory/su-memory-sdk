# su-memory SDK 技术迭代规划书 — v3.1.0 → v3.3.0

> **版本**: v1.0  
> **日期**: 2026-05-28  
> **来源**: 两轮 SOTA 基准测试（SDK 性能 + 记忆引擎 7 维度）的综合洞察  
> **当前版本**: v3.0.0 · Score 0.986 (A+)  
> **规划范围**: v3.1.0（性能修复）→ v3.2.0（能力增强）→ v3.3.0（规模演进）

---

## 一、产品定位

su-memory SDK v3.0.0 已在两轮 SOTA 基准测试中完成全面验证：

- **SDK 性能**：97K/s 插入吞吐、0.22ms P95 查询、0.33MB/1K 内存密度（4/4 回归 A+）
- **记忆引擎**：7 维度综合 0.986 (A+)，6 维满分，在所有可对比维度领先 SOTA baselines

测试同时暴露了三个结构性改进空间，按优先级排序：

1. **查询最差路径**：当数据集存在高频共享 token 时，查询从 O(log n) 退化到 O(n) 全扫描（D5 场景：450ms/查询）
2. **分词器数字失明**：阿拉伯数字被过滤，导致含数字标识的记忆不可区分（影响 D1 意译召回、D7 查询一致性）
3. **语义召回天花板**：纯 TF-IDF 关键词匹配在 paraphrase 场景下仅 66.7%，引入轻量 embedding 重排可突破此瓶颈

本规划书聚焦上述三个方向的工程实现，不改动已验证稳固的核心架构（索引结构、持久化、存储后端抽象）。

## 三、技术可行性评估

### 3.1 IDF 阈值剪枝 + heap top-K（v3.1.0）

| 维度 | 评估 |
|------|------|
| **实现复杂度** | 极低 — `query()` 方法中增加 2 行 IDF 跳过逻辑 + 1 行 `heapq.nlargest` 替换 |
| **性能收益** | D5 场景查询 450ms → <5ms（~90× 加速）；通用场景 4× 排序加速 |
| **召回影响** | 零 — IDF 剪枝仅跳过区分度为零的关键词（出现在 >50% 文档中） |
| **测试成本** | 低 — 运行现有 `sota_memory_engine.py` + `benchmark_v3.0.0.py` 回归 |
| **风险** | 几乎为零 — 纯局部优化，不改 API，不改数据结构 |

### 3.2 分词器数字保留（v3.1.0）

| 维度 | 评估 |
|------|------|
| **实现复杂度** | 中等 — `_tokenize()` 需重构中英文分离逻辑，将连续数字提取为独立 token |
| **性能收益** | D1 意译召回 66.7% → 预期 85%+；消除"数字失明"导致的区分度丢失 |
| **召回影响** | 正面 — 增加 token 多样性，提升精确匹配能力 |
| **测试成本** | 中等 — 需运行全部 7 维度 SOTA + 分词器单元测试 |
| **风险** | 中等 — 需验证：① token 数量不爆炸 ② 内存密度不显著劣化 ③ 向后兼容 |

### 3.3 Embedding 重排层（v3.2.0）

| 维度 | 评估 |
|------|------|
| **实现复杂度** | 高 — 新模块 `_semantic_reranker.py`，可选依赖 `sentence-transformers` |
| **性能收益** | D1 总分 0.900 → 预期 0.95+；意译召回 66.7% → 预期 90%+ |
| **内存代价** | all-MiniLM-L6-v2 模型 ~80MB，1K 条 @ 384-dim float32 ≈ 1.5MB 向量存储 |
| **测试成本** | 高 — 需验证双路检索（TF-IDF 召回 + embedding 重排）的召回提升和性能折衷 |
| **风险** | 中高 — ① 依赖安装失败降级 ② 嵌入计算延迟影响查询 P50 ③ 内存模型翻倍 |

## 四、分阶段实施计划

### M1: v3.1.0 — 查询性能消除最差路径（工期：3-5 天）

#### 子任务 1.1：IDF 阈值剪枝

**文件**：`src/su_memory/sdk/lite.py` — `query()` 方法  
**改动**：在候选集过滤循环中增加 IDF 跳过逻辑

```python
# 现有代码（lite.py L320-L327）
candidate_ids = None
for keyword in query_keywords:
    if keyword in self._index:
        if candidate_ids is None:
            candidate_ids = self._index[keyword].copy()
        else:
            candidate_ids &= self._index[keyword]

# 改为：
candidate_ids = None
total = len(self._memories)
for keyword in query_keywords:
    if keyword in self._index:
        df = len(self._index[keyword])
        # v3.1.0: IDF 阈值剪枝 — 出现超过50%的关键词无区分力，跳过
        if df > total * 0.5:
            continue
        if candidate_ids is None:
            candidate_ids = self._index[keyword].copy()
        else:
            candidate_ids &= self._index[keyword]
```

**预期效果**：D5 场景候选集从 5000 → <50，查询从 450ms → <5ms。

#### 子任务 1.2：heapq.nlargest 替代 sorted

**文件**：`src/su_memory/sdk/lite.py` — `query()` 方法 L361  
**改动**：

```python
# 现有代码：
sorted_ids = sorted(scores.items(), key=lambda x: x[1], reverse=True)

# 改为：
import heapq
sorted_ids = heapq.nlargest(top_k, scores.items(), key=lambda x: x[1])
```

**预期效果**：top-K 排序 O(n log n) → O(n log k)，k=3 时约 4× 加速。

#### 子任务 1.3：分词器数字保留

**文件**：`src/su_memory/sdk/lite.py` — `_tokenize()` 方法  
**改动要点**：

1. 在 `text_clean = re.sub(r'[^\u4e00-\u9fa5a-zA-Z0-9]', '', text.lower())` 后
2. 提取连续数字块：`re.findall(r'\d+', text_clean)`
3. 将每个数字块作为独立 token 加入 keywords（≥2 位数字时）
4. 中文 bigram 提取逻辑不变，但数字块已在 `chinese_chars = re.sub(r'[a-zA-Z0-9]', '', text_clean)` 中被剥离

```python
# 新增：提取数字 token（在现有中文分词之前）
digit_blocks = re.findall(r'\d+', text_clean)
for d in digit_blocks:
    if len(d) >= 2:
        keywords.add(d)
```

**预期效果**："订单号12345" → tokens 包含 "12345"，可区分于 "订单号54321"。

#### 子任务 1.4：Benchmark 前置验证

**文件**：`benchmarks/tokenizer_sanity.py`（新建）  
**内容**：验证分词器对 10 组典型中英数字混合输入产生预期 token

```python
def validate_tokenizer(engine):
    cases = [
        ("订单号12345", ["12345"]),       # 数字保留
        ("温度36.5度", ["36.5"]),         # 小数不保留（保持兼容）
        ("第0条和第6条", ["第0","第6"]),   # 数字结合中文
    ]
    for text, expected_tokens in cases:
        tokens = engine._tokenize(text)
        for et in expected_tokens:
            assert any(et in t for t in tokens), f"Missing '{et}' in {tokens}"
```

### M2: v3.2.0 — 语义召回增强 + 混合存储（工期：7-10 天）

#### 子任务 2.1：Embedding 重排层

**文件**：`src/su_memory/sdk/_semantic_reranker.py`（新建）  
**架构**：双路检索 — TF-IDF 粗排（top-20）→ embedding 精排（top-K）

```python
class SemanticReranker:
    """可选依赖 sentence-transformers，不可用时静默降级"""

    def __init__(self, model_name: str = "all-MiniLM-L6-v2"):
        self._model = None  # 懒加载

    def _ensure_model(self):
        if self._model is None:
            try:
                from sentence_transformers import SentenceTransformer
                self._model = SentenceTransformer("all-MiniLM-L6-v2")
            except ImportError:
                pass  # 静默降级

    def rerank(self, query: str, candidates: List[str], top_k: int) -> List[Tuple[int, float]]:
        if self._model is None:
            return [(i, 1.0) for i in range(min(top_k, len(candidates)))]
        q_emb = self._model.encode([query])[0]
        c_embs = self._model.encode(candidates)
        scores = [float(np.dot(q_emb, c_emb)) for c_emb in c_embs]
        top = heapq.nlargest(top_k, enumerate(scores), key=lambda x: x[1])
        return top
```

**集成点**：`SuMemoryLite.query()` 中增加可选参数 `semantic_rerank: bool = False`

#### 子任务 2.2：混合存储策略

**文件**：`src/su_memory/sdk/_tiered_storage.py`（新建）  
**策略**：

| 层级 | 存储 | 触发条件 | 容量 |
|------|------|----------|------|
| L0 热层 | 内存 `_memories` list | 默认 | ≤ `max_memories` |
| L1 温层 | SQLite `storage.db` | `count() > max_memories * 0.8` | 无上限 |
| L2 冷层 | PG/Redis | 显式配置 | 无上限 |

**自动降级**：当 L0 迫近 `max_memories` 时，最旧的 20% 记忆自动下沉到 L1。查询时三层联合检索，结果合并。

**覆盖**：10K / 25K / 50K 三个规模的插入吞吐、查询延迟、内存密度曲线

## 七、执行建议

1. **M1 优先执行，不改架构**：v3.1.0 的 IDF 剪枝 + heap top-K 是 3 行代码的局部优化，零风险、高回报（D5 查询 90× 加速）。数字保留分词器改动范围可控（约 10 行），但需充分回归。

2. **M2 双路检索需谨慎上线**：Embedding 重排是本次迭代中唯一可能显著改变查询行为的改动。建议先在 `SuMemoryLitePro` 中验证（其已有 embedding 基础设施），再决定是否下放到 `SuMemoryLite`。

3. **保留 `max_memories` 为性能保护阀**：当前插入 O(1) 的前提是 `len(self._memories) < max_memories`。M2 混合存储不应突破此约束——超限自动降级到 SQLite，而非在内存中无限扩展。

4. **每个里程碑都做全维度 SOTA 回归**：两轮 SOTA 证明了系统化基准的价值。将 `sota_memory_engine.py` + `benchmark_v3.0.0.py` 作为发布前的强制门禁。
